# 《Verify Before You Fix: Agentic Execution Grounding for Trustworthy Cross-Language Code Analysis》论文全解
该论文是2026年4月发布于arXiv的软件工程（cs.SE）领域预印本，由乔治华盛顿大学Jugal Gajjar完成，核心聚焦**可信智能体驱动的跨语言软件漏洞全生命周期分析**，提出了「先验证、再修复」的执行接地（Execution Grounding）核心原则，构建了检测-验证-修复三阶段闭环框架，在跨语言泛化、可解释性、端到端漏洞修复效果上取得了显著突破。

## 一、研究背景与核心问题
### 1.1 高风险场景下智能体流水线的本质缺陷
当前基于大语言模型（LLM）的学习分类器在代码理解、医疗诊断等领域表现突出，但部署于高风险场景时存在根本性可靠性问题：
- 模型输出是**基于训练分布的概率推断**，而非经过客观证据验证的结论；
- 下游智能体直接基于未验证的预测执行操作，会导致**错误逐级累积**：假阳性持续放大、下游资源浪费在非真实问题上、人工操作者因频繁误报丧失对系统的信任。

如何在多阶段 consequential 决策中处理不确定性、向人类合作者同步决策依据，是可信可解释AI的核心开放问题。

### 1.2 软件漏洞分析：理想的验证测试床
软件漏洞分析是验证上述问题的绝佳场景，具备两个关键特性：
1. **真值可验证**：漏洞的可利用性（exploitability）可以通过程序执行直接验证，能为AI推理提供客观反馈信号；
2. **天然多阶段**：完整漏洞处理包含「检测候选→确认可利用性→生成修复方案」三个阶段，分别对应结构模式识别、假设驱动执行推理、约束代码生成三种不同推理模态，非常适合研究多阶段智能体的可靠推理。

### 1.3 现有方案的核心短板
当前研究普遍将漏洞生命周期拆分为孤立工具：
- 检测模型只输出预测，不验证真实性；
- 修复系统直接基于未验证的检测结果生成补丁；
- 几乎所有方案都需要为每种编程语言单独训练模型，跨语言泛化能力差；
- 验证依赖静态分析而非真实执行，无法确认漏洞的实际可利用性；
- 可解释性依赖事后归因方法，稳定性差且额外增加推理成本。

## 二、核心贡献
论文提出了统一的跨语言漏洞生命周期框架，围绕「无执行确认则不执行修复」的严格不变量设计，核心贡献分为四点：

1. **面向跨语言LLM推理的通用结构表示**
   提出通用抽象语法树（Universal AST, uAST），实现Java、Python、C++的零样本跨语言迁移，跨语言F1达到74.43–80.12%，相比语言特定表示提升23.42%，且无需逐语言重新训练。

2. **带模态感知门控的可解释混合推理**
   通过双向门控机制融合结构特征与LLM语义特征，检测准确率达89.84–92.02%，同时每个预测自带样本级可解释性权重，无额外推理开销。

3. **作为可信AI机制的执行接地智能体验证**
   设计LLM驱动的验证智能体，将分类器的概率预测转化为执行验证后的确定性决策；消融实验显示，移除该阶段会导致不必要修复增加131.7%，端到端成功率下降9.56个百分点。

4. **支持人机协作的闭环迭代修复**
   仅对经过执行确认的漏洞启动修复，修复过程受重检测监督；对无法收敛的案例，输出结构化诊断轨迹移交人工审核，将人机协作作为一等设计原则。

## 三、框架整体架构
框架遵循「**推理→接地→行动**」的三阶段闭环原则，每个阶段都以前一阶段的验证证据作为准入条件，而非概率预测，该设计具备领域通用性，可迁移至所有预测可验证、误操作有可量化代价的场景。

针对漏洞分析场景，三个具体阶段为：
1. **混合结构-语义检测**：将源码解析为uAST，融合图嵌入与LLM语义嵌入，输出二元漏洞标记；
2. **执行接地的智能体验证**：对标记为漏洞的样本，通过沙箱执行生成的测试用例，确认真实可利用性；
3. **验证感知的迭代修复**：仅对确认可利用的漏洞生成补丁，通过重检测形成闭环，不收敛案例移交人工。

## 四、核心技术细节
### 4.1 通用抽象语法树（uAST）
跨语言漏洞检测的核心障碍是**表示不匹配**：同一种漏洞模式在不同语言中表层语法差异极大。uAST通过四层结构将Java、Python、C++归一化为统一结构模式：

1. **元数据层**：记录语言、节点总数、SHA-256内容哈希（用于去重）等全局属性；
2. **扁平节点数组**：将树形结构线性化为可O(1)寻址的节点列表，每个节点存储类型、文本内容、源码位置、父节点索引、子节点索引；
3. **通用节点分类法**：将200+种语言特定语法类型抽象为47个通用类别（如函数声明、变量赋值、控制流等），让Python的`def`、Java的`public static void`、C++的函数定义可以被统一查询；
4. **跨语言语义映射层**：将语言特有构造（Python的`with`、Java的try-with-resources、C++的RAII）映射为统一的通用角色，同时保留原始语法信息。

#### 实现方式
基于Tree-sitter构建（相比正则/规则方法，语法覆盖完整、跨语言接口统一、错误恢复能力强），流水线包含6个确定性阶段：语言检测→递归AST提取→节点分类→跨语言映射→模式验证→Parquet序列化。多核系统下平均每分钟处理600–800个文件。

### 4.2 混合检测模型：结构+语义双向门控融合
模型分为结构分支、语义分支，通过学习到的双向门控机制动态加权融合，同时天然具备可解释性。

#### （1）结构分支
将uAST编码为有向图，使用**GraphSAGE**进行图嵌入：
- 两层消息传递层，采用均值聚合，每个节点采样固定大小邻域；
- 最终输出128维的图级别嵌入；
- 加入拉普拉斯正则化，让连通图区域的嵌入更平滑。

#### （2）语义分支
使用**Qwen2.5-Coder-1.5B**处理原始源码：
- 对序列最后一层隐藏状态做位置平均，生成1536维语义嵌入；
- 超长文件采用滑动窗口处理，对分段嵌入取平均。

#### （3）双向门控融合机制
1. 将两种嵌入都做L2归一化，投影到128维共享隐空间，经过层归一化与激活函数处理；
2. 通过可学习的评分网络计算每个样本的模态得分，经softmax归一化为权重`[α_g, α_l]`（和为1）；
3. 加权融合得到最终表示，输入二分类器。

**可解释性**：权重直接反映决策依据——缓冲区溢出等结构明确的漏洞`α_g≈1`，需要语义上下文的逻辑漏洞`α_l≈1`，无需额外事后归因。

#### （4）复合损失函数
$$\mathcal{L}=\mathcal{L}_{CE}+\lambda_{nce } \mathcal{L}_{InfoNCE }+\lambda_{lap } \mathcal{L}_{Laplacian }$$
- 交叉熵损失：保证分类准确性；
- InfoNCE对比损失：促进两种模态的表示对齐；
- 拉普拉斯正则损失：平滑图结构上的嵌入。

### 4.3 执行接地的智能体验证
验证智能体实现**计划-执行-验证**推理循环，将概率性的检测输出转化为基于执行证据的二元判定。

#### 整体流程
1. **假设生成**：由Qwen2.5-Coder-1.5B-Instruct作为规划器，针对标记为漏洞的样本生成结构化利用假设，包括攻击向量、具体payload、环境前置条件；
2. **测试用例生成**：语言专属的测试生成器合成带插桩的测试程序；
3. **沙箱执行**：在Docker容器中隔离执行，严格限制资源（内存、CPU、网络、系统调用），设置超时；
4. **证据判定**：根据执行输出分类证据等级，给出最终可利用性结论；
5. **迭代优化**：初始测试无结论时，最多进行5轮迭代优化。

#### 语言专属插桩方案
| 语言 | 插桩与检测手段 |
|------|----------------|
| Python | 模块级猴子补丁：监控`sqlite3.Cursor.execute`、内置`open`、`os.system`/`subprocess`；捕获完整异常栈与局部变量 |
| Java | AspectJ字节码织入：监控JDBC查询、反射操作；SecurityManager限制文件/网络/本地库加载；JVMTI追踪污点传播 |
| C++ | 编译期插桩：AddressSanitizer检测内存安全问题、UBSan检测未定义行为；LD_PRELOAD拦截敏感库函数（`strcpy`/`system`/`fopen`等） |

#### 非对称执行策略
- **标记为漏洞（flag=1）**：主动验证，每个假设部署3–5个payload，60秒超时，目标是确认可利用性；
- **标记为安全（flag=0）**：有限探测，仅用1–2个通用payload，15秒超时，在控制开销的同时挽回部分假阴性；
- 触发确认利用立即提前终止，减少不必要执行。

#### 证据分级
分为确认性、提示性、中性三类，判定成功利用需要至少1项确认性证据。

### 4.4 验证感知的迭代修复
修复模块**仅对执行确认的漏洞激活**，避免在假阳性上浪费算力。

1. **补丁生成**：基于LoRA微调的Qwen2.5-Coder-1.5B-Instruct，输入包含漏洞代码、确认的漏洞类型、成功利用的payload、观测到的恶意行为，生成最小化补丁；
2. **语法校验**：通过语言专属AST操作库应用补丁（Python用RedBaron、Java用JavaParser、C++用libclang），保证语法合法且修改范围局部化；
3. **重检测闭环**：补丁后的代码重新解析为uAST，经过完整混合模型检测：
   - 检测为安全则终止，修复成功；
   - 仍检测为漏洞则带入历史上下文进入下一轮迭代；
   - 最多迭代5轮。
4. **人机协作**：耗尽迭代次数仍不收敛的样本，输出结构化诊断轨迹（所有尝试的补丁、被拒绝原因、残留漏洞指标），移交人工审核。

## 五、实验设计与结果
### 5.1 实验设置
#### 数据集
- 总规模：120,000个代码文件，60,000个漏洞样本（配对修复方案）+ 60,000个安全样本；
- 语言分布：Java、Python、C++各40,000个（各含20,000漏洞+20,000安全）；
- 划分：80%训练、10%验证、10%测试；
- 数据来源：80%来自15个真实漏洞数据库（PrimeVul、MegaVul、DiverseVul、CVEfixes等），20%由GPT-4.1与Claude Sonnet 4.5合成，保证SQL注入、命令注入、路径遍历、不安全反序列化等漏洞类型均衡覆盖。

#### 模型选型
对比3种图编码器（GCN、GraphSAGE、GAT）×3种LLM主干的9种组合，最终选择**GraphSAGE + Qwen2.5-Coder-1.5B**，验证集混合F1最高达90.21%。GraphSAGE普遍优于GCN和GAT，原因是其归纳式邻域聚合更适配uAST图不规则的节点度分布。

#### 运行环境
训练与主实验在Google Colab Pro的NVIDIA A100/H100 GPU上运行；本地推理可在Apple M4 MacBook Pro（48GB）上通过MLX框架执行。

### 5.2 检测阶段结果
#### 语内检测
混合模型在三种语言上准确率达**89.84–92.02%**，F1达0.8837–0.9109，比单模态基线高出8–15个百分点。
- C++准确率最高（92.02%）：内存管理漏洞的结构特征更明确，结构分支贡献更大；
- 结构与语义模态形成互补，融合后显著提升效果。

#### 零样本跨语言检测
| 训练→测试 | 仅结构F1 | 仅语义F1 | 混合F1 |
|-----------|----------|----------|--------|
| Java→Python | 0.5521 | 0.6824 | 0.7443 |
| C++→Java | — | — | 0.7818 |

整体跨语言F1达**74.43–80.12%**，相比无uAST的语言特定表示平均提升23.42%。纯结构模型跨语言效果接近随机，纯语义模型跨语言迁移能力差，证明**uAST结构归一化是跨语言泛化的核心驱动力**。

### 5.3 验证与修复阶段结果
#### 验证效果
| 语言 | 漏洞标记=1 | 漏洞标记=0 |
|------|------------|------------|
| | 可利用确认率 | 假阳性拒绝率 | 漏检挽回率 | 误报率 |
| Java | 70.28% | 60.43% | 13.87% | 2.94% |
| Python | 71.49% | 62.37% | 16.21% | 2.67% |
| C++ | 66.84% | 58.72% | 11.76% | 3.98% |
| 平均 | 69.54% | 60.51% | 13.95% | 3.20% |

- Python确认率最高：动态类型简化了自动化测试构造；
- C++确认率最低：内存级漏洞的触发条件更复杂；
- 对标记为安全的样本，有限探测仅带来极低的误报率，同时挽回了约14%的漏检漏洞。

#### 修复效果
| 语言 | 修复成功率 | 平均迭代轮数 | 最大迭代失败率 | 修复后检测通过率 |
|------|------------|--------------|----------------|------------------|
| Java | 85.14% | 2.7 | 10.38% | 91.36% |
| Python | 87.27% | 2.3 | 8.17% | 93.15% |
| C++ | 81.37% | 3.4 | 14.79% | 90.44% |
| 综合 | 83.78% | 2.8 | 11.12% | 91.63% |

- 修复难度排序：Python < Java < C++，与人工修复复杂度一致；
- 修复后检测通过率超90%，证明补丁有效消除了漏洞。

### 5.4 端到端性能与消融实验
#### 端到端整体表现
- 端到端漏洞解决率：**69.74%**；
- 总流水线失败率：12.27%；
- 验证阶段消除检测器假阳性比例：61.24%；
- 避免的不必要修复比例：73.13%；
- 单A100每日处理量：1700–2400个样本，单样本耗时37.4–49.3秒，可实际集成到CI/CD流程中。

#### 消融实验
| 配置 | 指标 | 数值 |
|------|------|------|
| 完整系统 | 漏洞解决率 | 69.74% |
| | 总失败率 | 12.27% |
| 消融A：移除uAST归一化 | 语内平均F1下降 | −3.79% |
| | 跨语言平均F1下降 | −23.42% |

消融实验直接量化了两个核心组件的必要性：
1. uAST几乎不影响语内性能，但对跨语言泛化至关重要；
2. 执行接地验证是避免无效修复、提升端到端可靠性的核心，移除后无效修复翻倍、成功率显著下降。

## 六、讨论与延伸价值
### 6.1 执行接地：通用的可信AI原则
论文通过消融量化了「无验证预测」的代价，提出**执行接地是多阶段LLM流水线降低不确定性的通用机制**：
- 不仅适用于漏洞分析，任何存在二元验证器的领域都可复用「计划-执行-验证」循环，包括定理证明、科学假设验证、带测试反馈的代码生成等；
- 核心逻辑：在预测与行动之间插入一个基于客观可观测结果的验证阶段，可系统性降低错误累积。

### 6.2 融合架构实现内在可解释性
双向门控的样本级权重天然提供可解释性，相比主流的事后归因（梯度、注意力）更稳定，不受输入扰动影响。这为多模态推理系统的可解释性提供了新思路：**在架构设计时嵌入可解释的加权机制，比事后解释黑盒模型更可靠**。

### 6.3 结构抽象是跨域迁移的关键
uAST带来的23.42%跨语言提升，验证了表示学习的通用规律：**与任务相关的结构抽象，能实现纯语义表示无法达成的迁移效果**。这类似于编译器中间表示实现跨硬件迁移、通用依存句法实现跨自然语言迁移的原理。

### 6.4 局限性与未来方向
#### 当前局限
1. 多跳推理视野有限：2层GraphSAGE不足以处理复杂的长距离污点流；
2. 沙箱环境重建不完整：复杂反序列化、格式化字符串漏洞易出现假阴性；
3. 评估覆盖不全：并发漏洞、密码学误用、业务逻辑漏洞尚未纳入测试。

#### 未来工作
- 用强化学习训练验证智能体，以漏洞利用确认作为奖励信号，提升复杂模式的发现率；
- 扩展语言覆盖：Tree-sitter支持40+语言，uAST模式已映射10种，扩展成本低；
- 集形式化验证：将补丁从「消除漏洞」升级为「可证明安全」，缩小自主AI与生产部署的差距。
